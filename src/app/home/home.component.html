Unit Testing is a way to document complex form logic

When looking up on how to use ReactiveForms in Angular, you usually come across some pretty simple examples. Essentially little more than how to hookup up the values to FormControls, configure in validators and maybe on how to use FormArrays and nested FormGroups.

But what about complex forms that have dynamic fields that are shown depending on form state? There can also be dynamic forms within FormArrays as well making the matter of controlling the application state reliably more difficult.

After development is completed on a complex form, the logic imprinted into the code is far from self-documenting. Without some strategy for documenting code, a developers ability to navigate and understand and code quickly after some time is diminished.


Unit Testing serves a dual purpose.

First, it provides a means of documentation within the code base so developers aren't context switching to external documentation like JIRA Stories, Confluence, Microsoft Office Products, etc...

Second, It tests the form logic... †A product of these tests are reports that ensure what was in the requirements is actually functioning as intended.

† Reports are only as good as the tests they are reporting. Without well though out and well written tests, reports are just pretty colors and jargon.
